Executive Summary

The application has moved from a Northwind-only Elasticsearch proxy toward a generic multi-source query platform. The frontend emits a shared AST, the API chooses a provider by domain, and providers compile the same request into Elasticsearch JSON or SQLite SQL.

The main unfinished Phase 3 work is analytical compilation for SQLite, richer UI support for range predicates, and relational JOIN-aware query planning beyond the current flattened AdventureWorks view.

Phase 3 Roadmap

1. SQLite Analytics Compiler

Implement ExecuteAnalytics by compiling ClientAggregation.Terms into GROUP BY queries and ClientAggregation.Sum into SUM(...). Keep the existing parameterized filter compiler and thread it into the analytics query as a reusable WHERE clause.

2. Range-Aware Visual Builder

The domain schema already exposes SupportsRange. The UI should swap the single text box for min/max numeric inputs when a range-capable field and range operator are selected.

3. Relational Query Planning

Replace the current AdventureWorks flattening shortcut with a field catalog that knows source tables, aliases, join paths, and allowed predicates. That is the path from "works for one view" to true relational portability.

4. Deployment Shape

Add container artifacts for the API, Elasticsearch dependency, SQLite volume, and compiled frontend assets. The current scripts are local-development oriented.

Recommended next task: start with SQLite analytics because it completes an existing user-facing tab without changing the shared AST or frontend contract.
